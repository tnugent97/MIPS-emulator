Overall results

There are three components to the tests shown here, which are extractions of the detailed results in the logs. The shading from green to red indicates values from 1.0 to 0.0, and the small values in brackets are the class average.

  • Ref Tester vs Your CPU : I run your CPU against my (assumed good) tester, and report what percentage of tests passed.
  • Your Tester vs Ref CPU : I run your tester against my (assumed good) cpu, and report what percentage of tests you claimed passed. The CPU is correct, so ideally all instructions should be shown to pass.
  • Your Tester vs Null CPU : I run your tester against a null CPU. The null CPU works perfectly, except for the single instruction you claim to be testing. For that instruction it will just return mips_Success and move onto the next instruction. It is almost impossible for a tester to show that it fails in all cases, but a good test-suite should have at least one test that shows it failing. Note that this means the linear scaling of colour is not quite right here - as long as there is one failed test the tester has done its job. So in the ideal case your tester would say that 0% of its instructions passed, but as long as it says <100% pass then it is doing it job.

The results of all the tests are included in the zip file, including the stderr, stdout, and csv files. I have included the results of running your tester on your cpu, but this isn't used for marking.

If you're seeing weird results above (and quite a few have them), then look in the relevant log files. For example, some people have tests that are applied to the ref CPU, but not to the null CPU (or vice versa). This usually indicates that your tester crashed or called exit, but only on one path.

Mapped grade

Your grades are extracted from the results using a blending function which is monotonic in success, i.e. each extra thing your CPU/tester does right increases marks. This gives a value from 0.0-1.0, which is then weighted as explained in the spec (I haven't done the bug report marks yet, as I need to check the map back from GH names). The correctness mark depends solely on how much manual work I had to do to get a submission working.

We are not allowed to give numeric marks, so I've given letter grades in the following ranges:
100-90A**
90-80A*
80-75A+
75-70A-
70-65B+
65-60B-
60-55C+
55-50C-
50-45D+
45-40D-
40-35E+
35-30E-
30-00F

So your marks are:

  • Correctness (20%): A**
  • Functionality (50%) : A**
  • Testing (30%): A**
  • Overall: A**

Note that the correctness component is at 100% unless I do anything, so significant or multiple modifications can decrease the overall mark by 20%.

Comments on overall marking

While this is all done a heavily automated workflow, the overall procedure is fairly manual. Sometimes people think my marking is all done at the push of a button, so just to reassure:

  1. Build all the submissions and check for compilation errors. Fix anything the crops up manually.
  2. Run all the CPUs against a reference tester. This throws up a bunch of errors such as crashes, that are then fixed.
  3. Run all the testers against a reference CPU. This throws up a bunch of errors including other crashes, that are then fixed.
  4. Once everything is built and running, check through all the output files for any submission that has an abormally low number of passing instructions, or where all tests fail. This often throws up something like little-endianness, not returning values from a function, not initialising variables, etc. which I'll fix in order to allow more tests to pass. One of the data-points used is the output of the self-test, as if there are lots of passing tests in the self-test, but not in other tests, it may indicate a deeper problem.
  5. Run all the final tests, including cross products.
  6. Look for anything strange again, e.g. zero-output, or cpus that assume they'll be run from a particular directory (for tests they can assume that).
  7. Generate all the final zips and do a final check.
This is less to convince you that I'm doing lots of work, and more to explain that if you have got a low result then I have looked at it and pondered it. If it was reasonable to update/change it then I would have done it in exchange for a penalty. But if it is only going to result in a minor improvement in results, or the underlying cause is that it is a low-functionality/rushed implementation, then I'll leave it alone.

If you're at the very top end, then I have looked through the top few implementations, and checked off the few failures. As far as I can tell, the implementations that are 100% correct on CPU and tester are correct, and the minor errors in CPU and/or test are genuine errors, based on the log outputs.
